KitBridgeunified IRT drug inventory · project overview

One clean, blinded view of clinical-trial drug inventory — across every IRT system.

Vendor-neutral middleware that reads from all the incompatible systems a research pharmacy juggles and turns them into a single standard the pharmacist can actually use.

The problem

Every sponsor uses a different IRT system. The pharmacy pays the price.

A research pharmacy runs many trials at once. Each trial’s sponsor tracks its investigational drug (IP) inventory in a different IRT/RTSM system — each with its own login, data format, date conventions, and quirks.

🔑

Many logins

A separate portal and credential set for every study. Context-switching all day.

🧩

Incompatible data

The same fact — a kit’s expiry — appears as 2026-08-15, 08/15/2026, or “Aug 2026”.

⚠️

Room for error

Manual reconciliation across systems risks the wrong kit, an expired kit, or a cold-chain miss.

IRT = Interactive Response Technology (a.k.a. RTSM) — the sponsor’s system of record for randomization and trial supply. IP = Investigational Product (the study drug).

The solution

A vendor-neutral layer that normalizes everything into one standard.

KitBridge sits between the messy vendor systems and the pharmacist. It ingests each vendor’s data, runs it through a normalizer, and presents one clean Kit object — consistent, searchable, and blinded.

  • One dashboard instead of N portals
  • One date format, one status vocabulary, one cold-chain label
  • Expiry warnings, study/status filters, and a full audit trail
  • Role-based visibility — pharmacist, monitor, admin
The core idea in one line:

Show the ugly inputs from three different vendors next to the one clean output — that contrast is the product.

Nobody in KitBridge ever sees active vs. placebo. Blinding stays intact; treatment assignment never leaves the sponsor backend.

Per-vendor adapters:

• cold-chain text → canonical label

Dates

08/15/2026 and “Aug 2026” both become 2026-08-15 / 2026-08-31.

Cold chain

“Refrigerate 2 to 8 degrees C” → 2-8°C; “Store frozen -20” → -20°C.

Gaps, surfaced

Vendor C omits storage/lot → the kit still loads, with warnings instead of silent blanks.

Two layers, on purpose

Vendor layer keeps the raw, messy payload exactly as received (auditable). Normalized layer is the clean product the app reads. The normalizer is the bridge.

Resilient by design

The app reads live from Supabase; if the database is unreachable it falls back to normalizing in-memory fixtures, so the demo always renders.

Roles & blinding

Everyone sees what they need — nobody sees the treatment arm.

👩‍⚕️ Pharmacist

Full blinded kit data + dispense logs. Day-to-day inventory operations.

🔎 Monitor

Limited verification info. Physical location & lot detail are hidden.

🛡️ Admin

Full audit trail across receiving and dispensing.

Blinded by design. KitBridge only ever stores and shows blinded labels (“Kit A”, “S-014”). Active-vs-placebo assignment never enters this system — it stays in the sponsor backend. That’s the integrity guarantee a trial requires.

Security & tech

Built on managed infrastructure, hardened from day one.

🔐 Security posture

  • Row-Level Security enabled on every table
  • Public anon key is read-only; RLS is the boundary
  • Service-role key kept server-side only, never shipped to the browser
  • Database security advisor: 0 issues

🧱 Stack

  • Frontend: Next.js (App Router) + Tailwind CSS
  • Backend: Supabase — Postgres, RLS, SQL migrations
  • Hosting: Vercel, auto-deploy on git push
  • Tests: Vitest — normalizer fully unit-tested

Deliberately out of scope for the prototype (post institutional backing): real IRT APIs, SSO, 21 CFR Part 11 compliance, e-signatures, and hospital-issued barcode scanners (BCMA). The phone-camera scan is demo-only.
